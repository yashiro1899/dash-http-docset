10    <h2 id="Syntax" name="Syntax">
11      Syntax
12    </h2>
13    <p>
14      The <code>data</code> URIs have the following syntax:
15    </p>
16    <pre class="eval">
17data:[&lt;mediatype&gt;][;base64],&lt;data&gt;
18</pre>
19    <p>
20      The <code>mediatype</code> is a MIME type string, such as "
 ><code>image/jpeg</code>" for a JPEG image file. If omitted, defau
 >lts to <code>text/plain;charset=US-ASCII</code>
21    </p>
22    <p>
23      If the data is textual, you can simply embed the text (usin
 >g the appropriate entities or escapes based on the enclosing docu
 >ment's type). Otherwise, you can specify <code>base64</code> to e
 >mbed base64-encoded binary data.
24    </p>
25    <p>
26      A few examples:
27    </p>
28    <dl>
29      <dt>
30        data:,Hello%2C%20World!
31      </dt>
32      <dd>

tt106    <h2 id="Common_problems" name="Common_problems">
107      Common problems
108    </h2>
109    <p>
110      This section describes problems that commonly occur when cr
 >eating and using <code>data</code> URIs.
111    </p>
112    <dl>
113      <dt>
114        Syntax
115      </dt>
116      <dd>
117        The format for <code>data</code> URIs is very simple, but
 > it's easy to forget to put a comma before the "data" segment, or
 > to incorrectly encode the data into base64 format.
118      </dd>
119      <dt>
120        Formatting in HTML
121      </dt>
122      <dd>
123        A <code>data</code> URI provides a file within a file, wh
 >ich can potentially be very wide relative to the width of the enc
 >losing document. As a URL, the <code>data</code> should be format
 >able with whitespace (linefeed, tab, or spaces), but there are pr
 >actical issues that arise <a class="external" href="http://bugzil
 >la.mozilla.org/show_bug.cgi?id=73026#c12">when using base64 encod
 >ing</a>.
124      </dd>
125      <dt>
126        Length limitations
127      </dt>
128      <dd>
129        Although Mozilla supports <code>data</code> URIs of essen
 >tially unlimited length, browsers are not required to support any
 > particular maximum length of data. For example, the Opera 11 bro
 >wser limits <code>data</code> URIs to around 65000 characters.
130      </dd>
131      <dt>
132        Lack of error handling
133      </dt>
134      <dd>
135        Invalid parameters in media, or typos when specifying "ba
 >se64", are ignored, but no error is provided.
136      </dd>
137      <dt>
138        No support for query strings, etc.
139      </dt>
140      <dd>
141        <p>
142          The data portion of a data URI is opaque, so an attempt
 > to use a query string (page-specific parameters, with the syntax
 > <code><em style="font-style:italic">&lt;url&gt;</em>?parameter-d
 >ata</code>) with a data URI will just include the query string in
 > the data the URI represents. For example:
143        </p>
144        <pre class="eval">
145data:text/html,lots of text...&lt;p&gt;&lt;a name%3D"bottom"&gt;b
 >ottom&lt;/a&gt;?arg=val
146</pre>
147        <p>
148          This represents an HTML resource whose contents are:
149        </p>
150        <pre class="eval">
151lots of text...&lt;p&gt;&lt;a name="bottom"&gt;bottom&lt;/a&gt;?a
 >rg=val
152</pre>
153        <p>
154          Note: as of Firefox 6, fragments (anchors) are processe
 >d consistent with other URI schemes, thus a bare "#" in the conte
 >nt needs to be escaped as '%23'.
155        </p>
156      </dd>
157    </dl>
158    <h2 id="Support_in_other_browsers" name="Support_in_other_bro
 >wsers">
159      Support in other browsers
160    </h2>
161    <p>
162      The <code>data</code> scheme is supported by Opera 7.20 and
 > above, as well as Safari and Konqueror. Internet Explorer 7 and 
 >below, however, do not currently support it. Internet Explorer 8 
 >and above only supports <code>data</code> URIs for images in CSS,
 > &lt;link&gt;, and &lt;img&gt;.
