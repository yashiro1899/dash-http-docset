Connection management is an important topic of HTTP: opening and maintening connections has a great impact on the performance of the Web site or the Web applications. In HTTP/1.x, there are several models: short-lived connections, persistent connections, and HTTP pipelining.

Most of the times, HTTP relies on TCP as the protocol transport. TCP provides a connection between the client and the server. In its enfancy, HTTP was provided one single model to handle these connections: they were short-lived as they were created each time a request needed to be sent, and closed once the answer had been received.

This simple model proved to hinder performance: opening a TCP connection is a time-consuming operation: there are several round-trip of messages exchanges between the client and the server, and the latency of the network was limiting the performance each time a request was sent. A modern Web pages needing numerous request (a dozen or more), this early model proved insufficient. Two more models have been added in HTTP/1.1.

HTTP/2 adds new models of connection management.

An important point to keep in mind is that connection management in HTTP applies to the connection between two consecutive nodes, that is hop-by-hop and not end-to-end. The model of connection between a client and its first proxy may be different then the one between this proxy and the destination server (or intermediate proxies). The HTTP headers involved in defining the connection model, like {{HTTPHeader("Connection")}} and {{HTTPHeader("Keep-Alive")}} are hop-by-hop headers and their values can be changed by intermediate nodes.

Short-lived connections

The original model of HTTP, and the default one in HTTP/1.0, is short-lived connections. Each HTTP request is done in its own connection; that means that there is a TCP handshake happening before each HTTP request, and these are serialized.

The TCP handshake itself is time-consuming, but a TCP connection adapts itself to its load and becomes more efficient with longer-lived connections. This model doesn't make use of this feature of TCP and performance is degraded from the optimum by always transmitting over cold connections, rather than over warm ones.

This model is the default model used with HTTP/1.0 (that is it is the one used if there are no {{HTTPHeader("Connection")}} header, or if its value is set to close. On HTTP/1.1, this model is used only if {{HTTPHeader("Connection")}} header is sent with a value of close.

Unless dealing with a very old system that doesn't support persistent connection, there is no compelling reason to use this model.

Persistent connections

Short-lived connection have two major drawbacks: the time to establish a new connection is significant, and the performance of the underlying TCP connection gets better when this connection has been in use for some time (warm connection). Even before HTTP/1.1, the notion of a persistent connection has been crafted, sometimes called a keep-alive connection.

A persistent connection is a connection that stays open for a long time and that can be reused for several requests, sparring the TCP handshake and making better use of the TCP performance features. The connection will not stay forever: idle connection are closed after some times (a server may use the {{HTTPHeader("Keep-Alive")}} header to give a minimum amount of time they should be kept open.

Persistent connections also have drawbacks, even idle they take some resources on the server and under heavy load, DOS can happen. In these cases, having a non-persistent connections, closed as soon as they are idle, improve performance.

HTTP/1.0 connection are not persistent by default. Setting {{HTTPHeader("Connection")}} to anything other than close, usually retry-after will make them by default. On HTTP/1.1, they are persistent by default and the header is not needed (but often added so that it is taken into account in case of a fallback to HTTP/1.0).

HTTP Pipelining

HTTP Pipelining is not activated by default in any modern browser for several reasons:

  • Buggy proxies are still common and these lead to strange and erratic behaviors that Web devs cannot foresee and diagnose easily
  • Pipelining is complex to implement correctly: the size of the object being transferred, the effective RTT that will be used, as well as the effective bandwith of the pipe have a direct incidence on the improvement given by pipelining. Without knowing these, important things may be delayed behing unimportant things and the notion of important evolves during page layout. That makes HTTP pipelining only a marginal improvement in most cases.
  • Pipelining is subject to the HOL problem.

Pipelining has been replaced by a better algorithm, multiplexing, that is used by HTTP/2. Use it instead.

Pipelining is the functionality to send several requests successively over the same persistent connection without waiting for the answer. That way the latency of the connection is avoided. Not all kind of HTTP requests can be pipelined: only {{glossary("idempotent")}} methods, that is {{HTTPMethod("GET")}}, {{HTTPMethod("HEAD")}}, {{HTTPMethod("PUT")}} and {{HTTPMethod("DELETE")}}, can be pipelined.

Every HTTP/1.1-compliant proxy and server should support pipelining, but in practice, a lot of them have limitations: that is one reason that no modern browser activate this feature by default.

Conclusion

Better connection management allowed to considerably improves performance of HTTP. When using HTTP/1.1 using a persistent connection, at least until it gets idle, leads to the best performance. Over the years, with the failure of pipelining, better connections managements have been designed and incorporated into HTTP/2.
