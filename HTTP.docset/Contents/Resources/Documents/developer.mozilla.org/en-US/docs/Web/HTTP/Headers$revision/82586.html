Revision Content

HTTP message headers are used to precise the resource being fetched or the behavior of the server or the client. Custom proprietary headers can be added using the 'X-' prefix; others are listed in an IANA registry whose original content was defined in RFC 4229. IANA also maintain a registry of proposed new HTTP message headers.

The following list summaries them and their usage:

  • Accept, see Content Negotiation
  • Accept-Charset
  • Accept-Control-*, see HTTP Access Control and Server Side Access Control{{ gecko_minversion_inline("1.9.1") }}, defined in the W3C Cross-Origin Resource Sharing

    • Access-Control-Allow-Origin
    • Access-Control-Allow-Methods
    • Access-Control-Allow-Headers
    • Access-Control-Max-Age
    • Access-Control-Expose-Headers
    • Access-Control-Request-Method
    • Access-Control-Request-Headers
    • Origin
  • Accept-Encoding
  • Accept-Language, see Content Negotiation
  • Accept-Ranges
  • Age
  • Allow
  • Authorization
  • Cache-Control, see HTTP Caching FAQ
  • Connection
  • Content-Encoding
  • Content-Language
  • Content-Length
  • Content-Location
  • Content-MD5  {{ unimplemented_inline("232030") }}
  • Content-Range
  • Content-Type, defines the MIME Type of the served document
  • Date
  • ETag, see HTTP Caching FAQ
  • Expect
  • Expires, see HTTP Caching FAQ
  • From
  • Host
  • If-Match
  • If-Modified-Since, see HTTP Caching FAQ
  • If-None-Match, see HTTP Caching FAQ
  • If-Range

  • Link, see Link Prefetching FAQ, defined in HTTP 1.1's RFC 2068, section 19.6.2.4
  • Location
  • Max-Forwards
  • Origin, see HTTP Access Control and Server Side Access Control{{ gecko_minversion_inline("1.9.1") }}, defined in the W3C Cross-Origin Resource Sharing

  • Proxy-Authenticate
  • Proxy-Authorization
  • Range
  • Referer

  • TE
