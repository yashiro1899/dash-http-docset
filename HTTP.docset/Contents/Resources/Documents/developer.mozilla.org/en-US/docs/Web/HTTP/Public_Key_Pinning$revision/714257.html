Revision Content

The Public Key Pinning Extension for HTTP (HPKP) is a security feature that tells the browser to associate a specific SSL certificate with a web site to prevent MITM attacks.

Note: The Public Key Pinning described here is different from the limited preload list based key pinning introduced in Firefox 32.

Enabling HPKP

Enabling this feature for your site is as simple as returning the Public-Key-Pins HTTP header when your site is accessed over HTTPS:

Public-Key-Pins: pin-sha256="base64=="; max-age=expireTime [; includeSubdomains]
pin-sha256
The quoted string is the the base 64-encoded SPKI (Subject Public Key Information) Fingerprint. It's possible to specify multiple pins and some browsers might allow other hashing algorithms in the future. See below on how to extract this information out of a certificate.
expireTime
The time, in seconds, that the browser should remember that this site is only to be accessed using the pinned certificates.
includeSubdomains {{ optional_inline() }}
If this optional parameter is specified, this rule applies to all of the site's subdomains as well.

Note: It's a good idea to include a pin for a backup certificate that isn't used in production. This ensures that the site stays accessible after the main certificate gets revoked.

 

Extracting the Base64 encoded public key information

First you need to extract the public key information from your certificate and encode them using base64.

Again, it is recommended to create a backup certificate that you are not using yet: By including the information for a future certificate, switch-over and revokation may work without breaking the pin, which is generally a good idea!

The following commands will help you extract the correct base64 encoded information for your certificate and your backup (which is most likely not a certificate, but just a certificate signing request). Change the filename in the following command and note that your file may also be in a different format than pem (e.g. der).

openssl x509 -inform pem -pubkey -noout < my-website.crt | openssl pkey -pubin -outform der | openssl dgst -sha256 -binary | base64
openssl req -inform pem -pubkey -noout < my-website-backup.csr | openssl pkey -pubin -outform der | openssl dgst -sha256 -binary | base64

We can now write the header as shown above with the correct base64.

Setting up your webserver to include the header

This depends highly on the web server you use.

Note: These examples use a high max-age (182.5 days) and include all sub domains. It is advised to verify that this setup works with your server.

Apache

Adding a line similar to the following will enable HPKP on your webserver. This requires apache mod_headers enabled on your server.

Header set Public-Key-Pins "pin-sha256=\"base64+info1==\"; pin-sha256=\"backup+pin+here==\"; max-age=15768000; includeSubDomains"

Nginx

Adding the following line and inserting the appropriate pin-sha256="..." values will enable HPKP on your webserver. This requires the ngx_http_headers_module.

add_header Public-Key-Pins 'pin-sha256="base64+info1="; pin-sha256="backup+pin+here=="; max-age=15768000; includeSubDomains';

Lighttpd

The following line with your relevant key information (pin-sha256="..." fields) will enable HPKP on lighttpd.

setenv.add-response-header  = ( "Public-Key-Pins" => "pin-sha256=\"base64+info1=\"; pin-sha256=\"backup+pin+here==\"; max-age=15768000; includeSubDomains")

Note: This requires the mod_setenv server.module loaded which can be included by the following if not already loaded.

server.modules += ( "mod_setenv" )
