Revision Content

In the Hypertext Transfer Protocol (HTTP), content negotiation is the mechanism used, when facing the ability to serve several equivalent content for a given URI, to provide the best suited one to the final user. The determination of the best suited content is either made through the use of specific HTTP headers by the client (server-driven negotiation), through the usage of the 300 Multiple Choices or 406 Not Acceptable HTTP response codes by the server (agent-driven negotiation) or through the use of a cache (transparent negotiation).

Server-driven negotiation

In this kind of negotiation, the browser (or any other kind of agent) send alongside with the URI several HTTP headers. These headers describe the preferred choice of the user. The server use them as hints and an internal algorithm let him choose the best content to serve to the client. The algorithm is server-specific and not defined in the standard. See, for example, the Apache 2.2 negotiation algorithm.

The HTTP/1.1 gives an exhaustive list of the standard header that may be used in a server-driven negotiation algorithm ({{ httpheader("Accept") }}, {{ httpheader("Accept-Charset") }}, {{ httpheader("Accept-Encoding") }}, {{ httpheader("Accept-Language:") }} and {{ httpheader("User-Agent") }}). Nevertheless it allows the server to use other aspects in its algorithm, either aspects outside the request itself, or extension header fields, i.e. headers not defined in the HTTP/1.1 standard.

The Accept: header

Defined in the HTTP/1.1 Standard, section 14.1, the Accept: header lists the MIME Types of the media that the agent is willing to process. It is comma-separated lists of MIME type, each combined with a quality factor, as parameter giving the relative degree of preference between the different MIME Types lists.

The Accept: header is defined by the browser, or any other user-agent, and can vary according of the context. It is therefore different when fetching an document entered in the address bar or an element linked via an {{ HTMLElement("img") }}, {{ HTMLElement("video") }} or {{ HTMLElement("audio") }} elements. Neither the HTTP standard, nor the HTML one, define specific MIME Type to use in specific contexts.

The Accept-Charset: header

Defined in the HTTP/1.1 standard, section 14.2, this header is used to indicates to the server what character encoding are understood by the user-agent. Traditionally, it was set to a different value for each locale for the browser, e. g. ISO-8859-1,utf-8;q=0.7,*;q=0.7 for a Western European locale.

Considering that :
  • UTF-8 is now well-understood by all relevant user-agents,
  • the presence of the header increase the  configuration-based entropy exposed,
  • the presence of the header increase the data transmitted for each request
  • almost no site is using the value of this header for choosing content during the negotiation,
browsers started to stop sending this header in each request, starting with Internet Explorer 8, Safari 5, Opera 11 and Firefox 6. In absence of Accept-Charset:, servers can simply assume that UTF-8 and the most common characters sets are understood by the client.

The Accept-Language: header

Defined in the HTTP/1.1 standard, section 14.4, this header is used to indicates the language preference of the user. A different values is set according the language of the graphical interface but most browsers allows setting different language preferences.

This header, especially when user-modified, increase greatly the configuration-based entropy and may be used in HTTP fingerprinting of the user.

The User-Agent: header

Defined in the HTTP/1.1 standard, section 14.43, this header identifies the browsers sending the request. This string may content a space-separated list of product tokens and comments.

A product token is a name followed by a '/' and a version number, like Firefox/4.0.1. There may be as many of them as the user-agent wants. A comment is a free string limited by parenthesis. Obviously parenthesis cannot be used in that string. The inner format of a comment is not defined by the standard, though several browser put several tokens in it, separated by ';'.

Though there are legitimate uses of this header for selecting content, it is considered bad practice to rely on it to define what features are supported by the user agent. Instead try to use in priority feature-oriented object detection.
